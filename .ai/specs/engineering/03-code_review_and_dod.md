# 代码审查 (CR) 与完成定义 (DoD)

## 1. 完成定义 (DoD)
提交合并（PR/MR）前必须满足：
1. **需求对齐**：代码实现与原始需求或拆解步骤严格对齐，不夹带私货。
2. **契约遵守**：符合既有接口、数据模型、异常处理规范。
3. **可验证**：提供测试用例及验证证据（截图/日志等）。
4. **零警告**：无 Linter 报错，符合安全与性能标准。

## 2. 代码审查重点
- **警惕“需求漂移”**：确认代码妥协方案（Workaround）是否违背了项目初衷。
- **强制“文档同步 (反哺)”**：若代码实现与设计文档不符，必须同步更新源文档。**代码可妥协，文档须诚实**，否则拒绝合并。
- **垂直切片**：鼓励端到端的独立可验收交付，拒绝无法独立运行的半成品。

## 3. PR 模版建议
建议包含：背景/目标、主要变更、测试证据、遗留问题（或技术债/文档同步说明）。
